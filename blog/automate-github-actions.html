Automate Your Deployments: A Beginner's Guide to GitHub Actions

Gab · March 2026 · 7 min read

If you've ever manually uploaded files to a server via FTP, you know it's error-prone. One wrong click and your site is down.

Modern development uses Continuous Deployment (CD). This means every time you save your code and push it to GitHub, a server automatically builds your site, runs tests, and deploys it live.

GitHub has this feature built-in. It's called GitHub Actions.

The Concept

A "Workflow" is a YAML file in your repository (specifically in .github/workflows/). It tells GitHub what to do when an event happens (like a push to the main branch).

You define "Jobs" inside the workflow. A job is a series of "Steps".

Asking Claude to Write Your Workflow

Writing YAML by hand is tedious. Just ask Claude.

The Prompt:

I have a static HTML website in a repository.
I want to deploy it to GitHub Pages automatically whenever I push to the main branch.

Write a GitHub Actions workflow file (`deploy.yml`) that:
1. Checks out the code.
2. Sets up Node.js (if needed for build steps, e.g., tailwind or sass).
3. Deploys the `./dist` folder to the `gh-pages` branch.

Claude will generate something like this:

name: Deploy to Pages

on:
  push:
    branches: ["main"]

permissions:
  contents: read
  pages: write
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      - name: Setup Pages
        uses: actions/configure-pages@v4
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: '.'
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4

Wait, Where Do I Put This?

  1. In your project root, create a folder named .github.
  2. Inside that, create a folder named workflows.
  3. Inside that, create a file named deploy.yml.
  4. Paste the content.
  5. Commit and push.

Go to your repository on GitHub, click the Actions tab, and watch your workflow run. If it's green, your site is live.

Beyond Deployment

You can use Actions for more than just deploying:

  • Testing: Run your unit tests before deploying. If tests fail, stop the deployment.
  • Linting: Check your code for style errors.
  • Notifications: Send a Slack message when a deployment succeeds or fails.
  • Scheduled Tasks: Run a script every night at midnight (cron jobs).

It's like having a robot assistant for your code.

Stop doing manual work. Let GitHub Actions handle your deployments while you sleep.
